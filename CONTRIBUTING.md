# Contributor Guidelines

Welcome! Thank you for your interest in contributing to this project.

Voxel Sandbox Engine is an experimental open-source project focused on exploring the design and implementation of a modern, data-driven voxel engine. Contributions are welcome from anyone interested in engine architecture, graphics, data-oriented design, or related topics.

This project evolves incrementally and deliberately. Thoughtful discussions and well-reasoned proposals are highly valued.

---

## Issues

### Feature Requests

If you have ideas on how to improve the project or propose new features, you can open a GitHub issue.

Please make sure to:
- clearly describe the proposed feature or change
- explain the motivation behind it
- outline potential use cases or benefits

Feature requests may be labeled as `enhancement`, and their status will be discussed directly in the issue.

---

### Bugs

When reporting a bug or unexpected behavior, please include:
- steps to reproduce the issue
- the environment or platform used (if applicable)
- any relevant error messages, logs, or screenshots

Clear and reproducible bug reports help the project move forward more effectively.

---

### Wontfix

Some issues may be closed and labeled as `wontfix` if they are out of scope, conflict with the project vision, or are not aligned with current goals. When possible, a brief explanation will be provided.

---

## Contribution Workflow

### Open Issues

If you would like to contribute, start by browsing open issues labeled `help wanted` or `good first issue`.

You are encouraged to comment on an issue to ask questions or to let others know that you are working on it. This helps avoid duplicated efforts.

---

### Making Changes

1. Fork the repository.
2. Create a new branch from `main`.
3. Make your changes, keeping them focused and well-scoped.
4. Ensure that your changes are consistent with the existing project structure and documentation.
5. Open a pull request with a clear title and description explaining what was changed and why.

A project maintainer will review your pull request and may request changes or clarifications before merging.

---

## Code Guidelines

At this stage of the project, no strict or automated code style is enforced.

However, contributors are expected to:
- write clear and readable code
- avoid unnecessary complexity
- prefer explicit and maintainable solutions over clever ones

Language- or tool-specific guidelines may be introduced as the project matures.
